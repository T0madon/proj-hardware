### 1. A Arquitetura do Sistema (O "Big Picture")
O projeto foi desenhado em uma arquitetura **Cliente-Servidor local**, baseada em Internet das Coisas (IoT). O processamento foi descentralizado:
* **O Hardware (ESP32) é o Servidor de Dados:** Ele não toma decisões. Seu único papel é ler os sensores de forma confiável, limpar os ruídos matemáticos e servir esses dados para quem pedir.
* **A Interface Web (JavaScript) é o Cliente e o "Cérebro":** O navegador atua como o cliente que consome os dados do ESP32, faz as contas complexas, cruza com o banco de dados local e dita a regra de negócio (se está apodrecendo ou não).

### 2. A Comunicação e o Endereço IP
Se o professor perguntar: *"Como eles conversam? O que é aquele IP?"*

* **A Rede Local (WLAN):** O ESP32 e o computador rodam na mesma rede Wi-Fi. Quando o ESP32 se conecta ao roteador, o roteador atribui a ele um **IP Local** (ex: `192.168.1.15`). Esse IP é como o "endereço residencial" da fruteira dentro da rede da faculdade/casa.
* **A API REST:** O código em C++ sobe um servidor web minúsculo na porta 80 do ESP32, criando uma rota chamada `/dados`.
* **O Polling (O método de busca):** A interface web não fica esperando o ESP32 mandar os dados. Ela usa uma técnica chamada **HTTP Polling**. A cada 2 segundos, o JavaScript dispara uma requisição (usando a função `fetch`) para o IP do ESP32 perguntando: *"Como estão as leituras agora?"*. O ESP32 responde entregando um pacote **JSON** leve com os números.

### 3. O Tratamento de Dados (Como limpamos os ruídos)
Sensores analógicos (MQ e Célula de Carga) sofrem muita interferência eletromagnética. Se jogássemos a leitura bruta na tela, o gráfico ficaria tremendo. 
* **No ESP32:** Ele faz leituras a cada **1 segundo** e soma esses valores. Quando chega em **10 segundos**, ele divide pela quantidade de leituras para tirar a **Média**. É essa média "suavizada" que ele entrega no JSON. Isso garante estabilidade.

### 4. A Matemática da Lógica (A Razão Proporcional)
Se o professor perguntar: *"Como o sistema sabe que está apodrecendo independente da quantidade de frutas?"*

A sacada de engenharia aqui foi não usar o volume bruto de gás, mas sim a **Razão Gás/Massa**.
1.  **Inversão da Leitura:** O sensor MQ funciona reduzindo a resistência quando o gás aumenta. A interface inverte isso fazendo `1 / leitura do sensor`. Assim, a lógica fica natural para o usuário final: **quanto maior o número final, mais gás tem no ar**.
2.  **A Proporção:** A interface pega esse número do gás e **divide pelo peso em Kg**.
3.  **Resultado:** Temos a "Razão Gás/Kg". Se você colocar 1 banana ou um cacho com 10 bananas do mesmo lote, a emissão de gás subirá junto com o peso, e a *razão* se manterá equilibrada. Se essa razão começar a subir independentemente do peso, é o sinal claro de degradação.

### 5. A Lógica de Alertas (Como o JS toma a decisão)
A cada 2 segundos, quando o pacote JSON chega do ESP32, o JavaScript processa os dados em uma "cascata" de regras (de cima para baixo):
* **Regra 1 (Fruteira Vazia):** Se o peso for menor que 50 gramas, o sistema entra em *Standby* (cinza). Ignoramos leituras de gás para não dar falso positivo com a fruteira vazia.
* **Regra 2 (Segurança contra Fermentação):** O sistema tem um limite rígido para o Álcool (MQ-3). Se houver um pico de álcool, ele ignora todo o resto e dispara o alerta vermelho crítico de fermentação.
* **Regra 3 (O Limite Único do Usuário):** Se as regras acima passaram, ele olha para a **Razão Gás/Kg** (Etileno do MQ-135) e compara com o limite que o usuário cadastrou no perfil (aquele limite que definimos "de gás e ponto"). Se a razão calculada for **menor ou igual** ao limite do usuário, está Verde (Bom). Se **ultrapassar** esse limite, está Vermelho (Apodrecendo).

### 6. O Servidor, o CRUD e o Banco de Dados
Se o professor perguntar: *"Onde está hospedado o servidor backend? E o banco de dados?"*

Você pode explicar que a arquitetura foi otimizada para **Zero-Cost (Custo Zero)** e processamento na Borda (Edge/Client-side).
* **Não existe um servidor backend tradicional** (como Node.js, Python ou PHP) rodando na nuvem.
* A página HTML atua como uma **SPA (Single Page Application)**.
* O banco de dados do CRUD é o **LocalStorage** do próprio navegador. O JavaScript converte os perfis de frutas criados pelo usuário em texto (JSON) e salva direto na memória do browser. Isso significa que não precisa de internet para funcionar (apenas o Wi-Fi local), garantindo resposta rápida e sem custos de hospedagem de banco de dados para a aplicação doméstica.
